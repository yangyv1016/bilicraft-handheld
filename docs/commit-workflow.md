# 提交与发布流程

本文件规定本仓库的**日常提交**与**发布 Release** 两条流程。以后每次提交都按这里执行。

## 核心原则（先记住这一条）

> **日常提交不写更新公告。只有在确认要构建 Release、打 `v*` tag 时，才编写面向玩家的更新公告。**

原因：更新公告是给**玩家**看的，只在发版那一刻才有意义。日常 commit message 是给**开发者**看的工程记录，两者受众不同、时机不同，不要混为一谈。App 内「检查更新」弹窗展示的公告，只取 GitHub Release 描述里**手写的那段**（详见下文《更新公告怎么写》）。

---

## 一、日常提交流程

平时开发、修 bug、重构，都走这条，**全程不涉及更新公告**。

1. 在 `master`（或从其派生的功能分支）上开发。
2. 只暂存与本次改动相关的文件，避免把无关变更一起提交。
3. 按下方《提交信息规范》写 commit message。
4. 推送。`ci.yml` 会自动跑 `assembleDebug` 验证可构建。
5. **不要**在日常提交里修改 `versionCode` / `versionName`，也**不要**写任何“更新公告 / 玩家公告”。

### 提交信息规范

沿用仓库既有风格：**英文类型前缀 + 冒号 + 空格 + 中文正文**。

```
<type>: <中文描述本次改动>
```

常用 `type`：

| 前缀 | 用途 |
| --- | --- |
| `feat` | 新增功能 |
| `fix` | 修复缺陷 |
| `refactor` | 重构，不改变外部行为 |
| `ci` | CI / 构建流水线相关 |
| `docs` | 文档 |
| `chore` | 杂项（依赖、脚本、配置） |
| `test` | 测试 |

示例（取自本仓库历史，保持一致）：

```
fix: 修正插件市场发现资产名
feat: 支持协议级命令补全
refactor: 服务器信息条瘦身为单行卡片
ci: 优化 release APK 产物命名
```

> 注意：历史上有过 `feat: 发布 1.0.0` 这类把「发布」写进日常提交的写法。**从现在起不再这样做**——发布动作由打 tag 承载（见下文），日常提交只描述代码改动本身。

---

## 二、发布 Release 流程（只有这里才写更新公告）

确认要正式发一个版本给玩家时，才走这条。

### 步骤

1. **升版本号**：改 `app/build.gradle.kts`：
   - `versionName`：玩家可见的版本号，如 `1.0.1`。
   - `versionCode`：每次发布**必须递增**的整数（App 内更新判断依赖它，见 `UpdateManager.isRemoteNewer`）。
   - 单独提交，例如：`chore: 升版本到 1.0.1`。
2. **打 tag 并推送**：tag 必须是 `v` + 版本号，与 `versionName` 对应。

```bash
git tag v1.0.1
git push origin v1.0.1
```

3. 推送 tag 会触发 `release.yml`：构建**已签名** `assembleRelease` → 产物挂到 GitHub Release，并自动生成开发者 changelog。
4. **编写更新公告**：到 GitHub 上刚创建的这个 Release，**编辑描述，在最顶部写给玩家看的公告**（怎么写见下一节）。保存即可，App 端下次检查更新就会读到。

### 版本号与 tag 的对应关系

| 位置 | 值 | 说明 |
| --- | --- | --- |
| `versionName` | `1.0.1` | 玩家可见版本号 |
| git tag | `v1.0.1` | 触发 release，`v` + versionName |
| `versionCode` | `14` | 递增整数，决定「是否有新版本」 |

三者必须同步。`versionCode` 忘记递增会导致老用户收不到更新提示。

---

## 三、更新公告怎么写

App 内「检查更新」弹窗**不会**展示 GitHub 自动生成的 changelog（那是给开发者看的 PR/commit 列表）。它只截取 Release 描述里 **`## What's Changed` 之前**、由你手写的那段，逻辑见 `ReleaseNotes.humanize`。

因此：GitHub Release 描述的结构天然是——

```
（你手写的玩家公告，写在最顶部）      ← App 只展示这部分

## What's Changed                    ← GitHub 自动追加，App 自动丢弃
* xxx by @someone in #12
...
**Full Changelog**: https://...
```

### 写作要求

- **写在最顶部**，在 `## What's Changed` 自动段之前。
- **面向玩家，讲人话**：说清楚这版「新增了什么、修了什么、玩家能感知到的变化」，不要贴 commit 号、PR 链接、内部模块名。
- 支持基础 markdown（`**加粗**`、`[文字](链接)`、列表），展示时会被清洗成纯文本。
- **不写会怎样**：`humanize` 取不到手写正文时，会兜底展示「本次更新包含若干优化与修复，建议更新到最新版本。」。App 不会崩，但等于没给玩家有效信息，所以正式发版**务必手写**。

### 示例

```markdown
本次更新优化了服务器会话界面，聊天输入框不再被按钮遮挡；
连接状态改用彩色圆点更直观。同时修复了自动补全后光标错位的问题。

- 服务器信息卡片瘦身，聊天区域更大
- 插件入口移到顶部，不再挡住输入框
- 点击通知栏可直接回到应用
```

## 四、CI / CD 流水线速查

| 流水线 | 触发 | 作用 | 与公告的关系 |
| --- | --- | --- | --- |
| `ci.yml` | push / PR 到 `main`·`master` | `assembleDebug` 验证可构建 | 无关，日常提交自动跑 |
| `release.yml` | 推送 `v*` tag | 构建已签名 `assembleRelease` + 建 GitHub Release | **这里之后才手写公告** |
| `deploy-cdn.yml` | 手动 / 定时 / Release 成功后 | 生成并部署 CDN 索引（更新源、插件市场、CDK） | 依赖 Release 已发布 |

---

## 一句话总结

- **日常**：改代码 → `<type>: 中文描述` → 推送。不碰版本号，不写公告。
- **发版**：升 `versionCode`/`versionName` → 打 `v*` tag 推送 → 在 GitHub Release 顶部手写玩家公告。
